NIP-340 schlägt Konventionen für die Nutzung von [FROST](/de/topics/frost/) (Flexible Round-Optimized Schnorr Threshold) auf Nostr vor. FROST erlaubt einer Gruppe von Signern, eine Nostr-Identität gemeinsam zu kontrollieren, sodass beliebige t-von-n Mitgliedern Events signieren können, ohne den vollständigen private key zu rekonstruieren.

## Funktionsweise

Ein FROST-Quorum verteilt Key Shares auf n Teilnehmer. Beliebige t dieser Teilnehmer können zusammenarbeiten, um eine gültige Schnorr-Signatur für den gemeinsamen Nostr-pubkey zu erzeugen. Kein einzelner Teilnehmer hält jemals den vollständigen private key, und weniger als t Teilnehmer können nicht signieren. Das NIP definiert, wie Signing-Runden koordiniert, Key Shares verteilt und threshold-signierte Events über Nostr-Relays veröffentlicht werden.

## Anwendungsfälle

- **Organisations-Accounts**: Ein Projektteam kontrolliert eine gemeinsame Nostr-Identität, ohne dass ein einzelnes Mitglied vollständige Signierautorität besitzt
- **Custody**: Multi-Party-Signing für sensible Operationen
- **Release-Verifikation**: Dezentrale Signierung von Software-Release-Events
