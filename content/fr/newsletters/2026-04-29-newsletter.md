## Articles principaux

### GitWorkshop livre la fusion de PR dans le navigateur, le suivi de dépôts et un explorateur git économe en bande passante

[GitWorkshop](https://gitworkshop.dev), la couche de collaboration web de Dan Conway pour [NIP-34](/fr/topics/nip-34/) git-over-Nostr, a livré une sortie majeure cette semaine qui rapproche beaucoup le workflow de ce que les développeurs attendent de GitHub ou GitLab, tout en gardant les commentaires, les listes de dépôts et les notifications à l'intérieur d'événements Nostr signés.

L'ajout phare est un bouton de fusion de PR dans le navigateur attendu depuis longtemps pour les dépôts utilisant des relais GRASP. La sortie ajoute également les Étoiles et le suivi de dépôts construits sur les réactions et les listes [NIP-51](/fr/topics/nip-51/), avec des ensembles de dépôts épinglés publiés en tant qu'événements de kind `10617` qui pointent vers des annonces de dépôts de kind `30617` via des balises `a` ordonnées. Les pages de profil peuvent maintenant présenter une liste portable de dépôts.

Un explorateur git économe en bande passante remplace le clone superficiel précédent dans le navigateur. Le nouvel explorateur s'appuie sur le protocole client/serveur git sous-jacent sur lequel GRASP se construit, il peut donc gérer de grands dépôts sans forcer le navigateur à récupérer un pack complet. La recherche couvre maintenant les noms d'utilisateur et les métadonnées de dépôt, alimentée par [NIP-50](https://github.com/nostr-protocol/nips/blob/master/50.md) et une implémentation de relais `ngit-indexer` qui découvre et synchronise les annonces de dépôts à travers le réseau. Un workflow de création de dépôt dans le navigateur complète le chemin de découverte et d'onboarding.

L'outillage de revue est reconstruit autour d'un onglet Fichiers modifiés, d'un visualiseur de diff par patch, et d'un ensemble de nouvelles primitives expérimentales. Les commentaires de revue de code en ligne utilisent le kind `1111`, construit sur [NIP-22](https://github.com/nostr-protocol/nips/blob/master/22.md) : chaque commentaire pointe vers un chemin de fichier (balise `f`), un SHA de commit (balise `c`), et une plage de lignes sélectionnées (balise `line`) afin qu'un client puisse rendre le commentaire à la bonne position dans un diff. Un deuxième niveau de primitives expérimentales est autorisé par l'auteur et les mainteneurs du dépôt et utilise les étiquettes [NIP-32](https://github.com/nostr-protocol/nips/blob/master/32.md) : renommer un sujet d'Issue ou de PR après soumission, ajouter des hashtags après soumission, épingler une CoverNote versionnée en haut d'une PR ou d'une Issue pour un résumé modifiable, et marquer les sous-fils de discussion de code en ligne comme résolus. Les événements Verdict et les blocs `suggestion` restent en projet et n'ont pas encore été livrés.

L'état des notifications entre appareils est également synchronisé via Nostr, mais avec une touche préservant la vie privée. GitWorkshop génère une paire de clés dédiée aux notifications, chiffre cette nsec et la stocke à l'intérieur d'un événement de kind `30078`. La nsec des notifications signe ensuite les événements d'état de notification réels. L'indirection empêche le signataire principal de l'utilisateur d'être spammé avec des demandes fréquentes de chiffrement et déchiffrement pour chaque action de lecture ou d'archivage, et elle empêche les observateurs extérieurs de voir facilement quand un utilisateur touche à son état de notification. Un utilisateur peut synchroniser l'état de lecture et d'archivage entre les appareils ; les relais ne voient que des blobs chiffrés.

### Routstrd lance un routeur local pour l'inférence sur Nostr

[Routstrd](https://github.com/routstr/routstrd) est un nouveau démon TypeScript qui donne aux outils locaux un point de terminaison compatible OpenAI et route chaque requête vers un fournisseur [Routstr](https://routstr.com) concurrent. Le démon découvre les fournisseurs via des annonces Nostr de kind `38421` définies dans la spécification RIP-02 de Routstr. Il note ensuite les fournisseurs selon le prix, la confiance et les performances récentes sous RIP-06 et envoie chaque requête à la meilleure option actuelle.

Le paiement passe par un portefeuille Cashu local géré par cocod et alimenté avec Lightning. Cela donne au client un chemin de règlement dénommé en sats tout en gardant la découverte des fournisseurs publique et sans permission via les relais Nostr. Si un fournisseur échoue pendant une session, Routstrd peut se rabattre sur le nœud classé suivant. Le chemin d'installation est `bun install -g routstrd`, suivi de `routstrd onboard` pour la configuration du portefeuille et des relais.

L'organisation [Routstr](https://github.com/routstr) plus large maintient le démon, le logiciel de nœud Python (`routstr-core`), une UI de chat et les spécifications de protocole. Pour les utilisateurs, le port local devient l'interface stable : les outils compatibles OpenAI existants pointent vers Routstrd, tandis que le démon gère la découverte des fournisseurs, le routage et le paiement.

## Sorties étiquetées

### ngit v2.4.2 corrige la détection du relais GRASP pour les soumissions de PR

[ngit](https://codeberg.org/DanConwayDev/ngit-cli) a livré [v2.4.2](https://github.com/DanConwayDev/ngit-cli/releases/tag/v2.4.2) avec un correctif pour la détection du serveur GRASP du dépôt, gardant la soumission de PR sur le chemin heureux lorsqu'une proposition utilise le kind PR. Notez que ngit utilise actuellement par défaut le kind `Patch` pour la plupart des changements sauf s'ils sont volumineux ; le mainteneur travaille à changer le défaut. [v2.4.1](https://github.com/DanConwayDev/ngit-cli/releases/tag/v2.4.1), livré plus tôt dans la semaine, a corrigé les erreurs `fatal` lors du clone et du fetch quand les données git d'une PR ouverte n'étaient pas disponibles sur les serveurs git spécifiés par le dépôt.

### Wisp v1.0.0 sort de la bêta

[Wisp](https://github.com/barrydeen/wisp), un client Android Kotlin et Jetpack Compose axé sur le routage des relais, la vie privée et une petite UI native, a livré [v1.0.0](https://github.com/barrydeen/wisp/releases/tag/v1.0.0) et enchaîné avec [v1.0.2](https://github.com/barrydeen/wisp/releases/tag/v1.0.2). Le jalon 1.0.0 rassemble le bascule de dénomination fiat Normie Mode, le fil Pour Vous, la configuration de groupe basée sur les relais [NIP-29](/fr/topics/nip-29/) et la diffusion de listes de relais [NIP-65](/fr/topics/nip-65/) couvertes dans [Newsletter #19](/fr/newsletters/2026-04-22-newsletter/#wisp-v0180-beta-adds-normie-mode-for-you-feed-and-nip-29-group-config). v1.0.2 ajoute le support de taille de page 16 Ko d'Android 15, un onglet de scan QR dans le tiroir, un bouton de téléchargement pour les contrôles vidéo en ligne et des correctifs de performance de la liste de notifications.

### grain v0.5.2 corrige le blocage WebSocket, v0.5.3 poursuit le polissage

[grain](https://github.com/0ceanSlim/grain), le relais Go de 0ceanSlim, a coupé [v0.5.2](https://github.com/0ceanSlim/grain/releases/tag/v0.5.2) comme correctif critique pour un blocage WebSocket introduit dans v0.5.0, puis a enchaîné avec [v0.5.3](https://github.com/0ceanSlim/grain/releases/tag/v0.5.3). Le blocage causait le figeage des connexions sous certains chemins de filtre et WebSocket, les opérateurs sur v0.5.1 ou v0.5.0 devraient donc mettre à niveau. grain suit toutes les catégories majeures d'événements Nostr, expose les informations de relais NIP-11, prend en charge le contrôle d'accès par liste blanche/noire, les limites de taux par kind, un tableau de bord web et une bibliothèque cliente Go ajoutée dans la ligne v0.5.x.

### Mostro Core v0.10.0 et Mostro Mobile v1.2.5 adoptent la double clé gift wrap NIP-59

[Mostro Core v0.10.0](https://github.com/MostroP2P/mostro-core/releases/tag/v0.10.0) ajoute le nouveau module gift-wrap [NIP-59](/fr/topics/nip-59/) avec identité séparée et clés de trade. Le code de transport précédent utilisait une clé d'identité unique à la fois pour l'identité de trade et pour le gift wrapping. v0.10.0 sépare l'identité de trade stable de la clé de wrapping éphémère, afin que chaque trade puisse utiliser une nouvelle clé de transport tout en préservant l'identité nécessaire au protocole de trade. L'intégration du démon arrive via [Mostro PR #718](https://github.com/MostroP2P/mostro/pull/718), et [mostro-cli PR #165](https://github.com/MostroP2P/mostro-cli/pull/165) apporte la même migration au client en ligne de commande.

[Mostro Mobile v1.2.5](https://github.com/MostroP2P/mobile/releases/tag/v1.2.5) est livré aux côtés du travail de protocole. [PR #581](https://github.com/MostroP2P/mobile/pull/581) permet aux preneurs de filtrer les offres selon l'âge du compte du maker, donnant aux utilisateurs un moyen d'éviter les comptes maker nouvellement créés dans le carnet d'ordres. [PR #580](https://github.com/MostroP2P/mobile/pull/580) corrige les étiquettes de rôle sur les détails des commandes annulées, et [PR #576](https://github.com/MostroP2P/mobile/pull/576) nettoie les boutons d'annulation coopérative.

### marmot-ts v0.5.0 livre des KeyPackages adressables

[marmot-ts](https://github.com/marmot-protocol/marmot-ts) a coupé [@internet-privacy/marmot-ts@0.5.0](https://github.com/marmot-protocol/marmot-ts/releases/tag/%40internet-privacy%2Fmarmot-ts%400.5.0), la première sortie planifiée avec changements incompatibles pour le client TypeScript [Marmot](/fr/topics/marmot/). [PR #68](https://github.com/marmot-protocol/marmot-ts/pull/68) ajoute le support des KeyPackages adressables : `KeyPackageManager` peut maintenant gérer à la fois les événements KeyPackage legacy de kind `443` et les nouveaux de kind `30443`. La sortie supprime `KeyPackageStore` et les classes de stockage d'état de groupe, en les remplaçant par des stockages clé-valeur génériques passés à `KeyPackageManager` et `MarmotGroup`. Elle déplace également la gestion des invitations et des groupes sur `MarmotClient.invites` et `MarmotClient.groups`, donc les intégrateurs directs ont besoin de changements de constructeur et de stockage avant la mise à niveau.

### CruxCoach v0.1.3 livre une sauvegarde chiffrée des données d'escalade avec Nostr et Blossom

[CruxCoach](https://codeberg.org/CruxCoach/CruxCoach) est une nouvelle application Android open source pour les grimpeurs de Kilter Board. La Kilter Board est un mur d'entraînement interactif dont les prises s'allument via Bluetooth pour afficher les voies. L'application a été lancée le 14 avril et a atteint [v0.1.3](https://codeberg.org/CruxCoach/CruxCoach/releases/tag/v0.1.3) le 26 avril.

v0.1.3 ajoute la sauvegarde cloud chiffrée en opt-in. Le compte CruxCoach d'un utilisateur est une paire de clés Nostr, et la clé privée sert aussi d'entrée à la clé de chiffrement de sauvegarde locale. L'application chiffre les données d'escalade sur l'appareil et miroite le chiffré sur les serveurs de stockage Blossom (`blossom.primal.net` et `nostr.download`). Les actions de suppression distante appellent le chemin de nettoyage Blossom. Au-delà de la sauvegarde, CruxCoach utilise la signature à distance [NIP-46](/fr/topics/nip-46/) pour le support d'Amber, les DM privés [NIP-17](/fr/topics/nip-17/) pour le contact avec le développeur dans l'application, les listes de relais [NIP-65](/fr/topics/nip-65/) pour la découverte des relais, et la bibliothèque [Quartz](https://github.com/vitorpamplona/quartz) de Vitor Pamplona pour la plomberie Nostr. Les utilisateurs peuvent l'installer via Zapstore ou les APKs Codeberg directs.

## Nouvellement suivis et découverts

### Nostrord : un client NIP-29 construit avec Kotlin Multiplatform et WASM

[Nostrord](https://github.com/nostrord/nostrord) est un nouveau client de chat de groupe [NIP-29](/fr/topics/nip-29/) ciblant le cas d'usage de remplacement de Discord. Les groupes vivent sur les relais Nostr avec l'adhésion, les rôles, la modération et le contrôle d'accès appliqués par le relais, donc l'état du groupe est hébergé par le relais NIP-29 sélectionné. Le développeur du client ne contrôle pas de base de données d'application séparée pour ces groupes. L'application web fonctionne sur [web.nostrord.com](https://web.nostrord.com) et est construite avec Kotlin Multiplatform compilant vers WebAssembly, avec des builds natifs Android, iOS et desktop en développement. Nostrord est bénéficiaire d'une subvention [OpenSats](https://opensats.org) et interopère avec les mêmes relais NIP-29 utilisés par Flotilla, Chachi et 0xChat.

### Clave apporte la signature à distance NIP-46 sur iOS via APNs

[Clave](https://github.com/DocNR/clave) est un signataire à distance iOS en bêta qui signe les événements Nostr quand l'application n'est pas ouverte. La clé privée reste dans le trousseau iPhone. Quand un client envoie une demande de signature à distance [NIP-46](/fr/topics/nip-46/), un proxy côté serveur délivre une Apple Push Notification, réveillant une Notification Service Extension pendant jusqu'à 30 secondes. Cette extension déchiffre la requête avec le chiffrement [NIP-44](/fr/topics/nip-44/), signe avec la clé du trousseau et publie la réponse. L'enregistrement du token de l'appareil utilise l'auth HTTP [NIP-98](https://github.com/nostr-protocol/nips/blob/master/98.md) pour empêcher le détournement de token. Clave prend en charge l'appairage `bunker://` et `nostrconnect://`, les niveaux de confiance par client, les remplacements par kind, et a été testé avec Nostur et noStrudel.

### Treasures : géocaching décentralisé sur Nostr

[Treasures](https://gitlab.com/chad.curtis/treasures) est une plateforme de géocaching où les caches et découvertes sont des événements Nostr signés. Les créateurs de cache publient des événements adressables de kind `37516` avec les coordonnées GPS. Les découvreurs enregistrent la découverte en scannant un code QR attaché à la cache physique ; le code encode le pubkey du créateur, la balise `d` de la cache et une clé privée de vérification utilisée comme preuve de la visite physique. Les zaps [NIP-57](/fr/topics/nip-57/) peuvent circuler des découvreurs aux créateurs de caches, et l'application en direct est sur [treasures.to](https://treasures.to).

### smesh v0.5.1 : relais Nostr, client et signataire auto-hébergés en un seul stack

[smesh](https://git.smesh.lol/smesh/smesh) est une pile Nostr auto-hébergée écrite en Moxie, un langage personnalisé dérivé de Go et TinyGo par mleku. La pile livre un binaire de relais natif avec support HTTP, WebSocket, AUTH, recherche et Blossom ; `sm3sh`, un client web compilé en modules ES ; et une extension de signature de navigateur avec signature de navigateur NIP-07 plus support de chiffrement NIP-04 et NIP-44. Le travail récent inclut la messagerie de groupe MLS (RFC 9420) dans v0.5.0, la réconciliation d'ensembles negentropy pour la synchronisation de relais, et un moteur de graphe Web of Trust. Le code vit sur la forge auto-hébergée de mleku à `git.smesh.lol`, construite avec son propre outil `git-web`. Le dépôt connexe [gitea-nostr-auth](https://git.smesh.lol/smesh/gitea-nostr-auth) est un pont OAuth2/OIDC pour Gitea : les utilisateurs s'authentifient avec un signataire de navigateur NIP-07, le pont découvre les relais via NIP-65, et Gitea reçoit des revendications d'identité OIDC standard.

### Surveil : un constructeur de deck Magic: The Gathering sur Nostr

[Surveil](https://gitlab.com/chad.curtis/surveil) est un client Nostr pour les joueurs de Magic: The Gathering qui permet aux utilisateurs de rechercher des cartes, de construire des decks, de scanner des cartes papier sur Android avec l'OCR ML Kit sur l'appareil, et de partager des decks à travers le réseau. Les decks sont publiés en tant qu'événements adressables de kind `37381`, et la spécification de l'événement de deck est documentée dans le `NIP.md` du projet. La couche sociale est construite à partir de primitives Nostr standard : commentaires en fil NIP-22 (kind `1111`) portés sur chaque deck, réactions NIP-25 (kind `7`), données de profil [NIP-78](https://github.com/nostr-protocol/nips/blob/master/78.md) (kind `30078`) pour les accueils de joueurs, fils de suivi de kind `3`, et forks qui portent une balise `a` de retour au deck original. [v0.1.6](https://gitlab.com/chad.curtis/surveil/-/tags/v0.1.6) a été livré cette semaine avec un polissage de l'UI mobile, des améliorations du compteur de vie, une page À propos revue, et une pastille de relais sur la bannière héros du deck. L'application web fonctionne partout où du HTML statique est servi, la version Android est livrée via [Zapstore](https://zapstore.dev), et les événements de kind `37381` sont également indexés nativement par [Ditto](https://about.ditto.pub/reference) comme decks Magic. Le dépôt est sur GitLab à [chad.curtis/surveil](https://gitlab.com/chad.curtis/surveil).

### Mises à jour NIP

Propositions et discussions récentes dans le [dépôt NIPs](https://github.com/nostr-protocol/nips) :

**Fusionnées cette semaine :**

- **NIP-34 dépôts git : suppression de l'extension de balise refs inutilisée** ([PR #2325](https://github.com/nostr-protocol/nips/pull/2325)) : Supprime une extension de balise `refs` de [NIP-34](/fr/topics/nip-34/) qui était définie mais inutilisée. Le nettoyage réduit l'ambiguïté d'implémentation pour les outils git-over-Nostr.

- **NIP-34 dépôts git : suppression de la revendication NIP-09 incorrecte** ([PR #2326](https://github.com/nostr-protocol/nips/pull/2326)) : Supprime une revendication incorrecte selon laquelle les événements de suppression [NIP-09](https://github.com/nostr-protocol/nips/blob/master/09.md) peuvent réinitialiser l'état du dépôt. La suppression NIP-09 est une demande de suppression d'événement côté client, pas une machine d'état de dépôt. La correction empêche les implémenteurs NIP-34 de traiter les indices de suppression comme des réinitialisations de dépôt faisant autorité.

**Travail ouvert et piloté par l'implémentation :**

- **Commentaires de revue en ligne de kind `1111` de GitWorkshop** : Le kind de commentaire de revue de code en ligne est documenté dans le `NIP.md` de GitWorkshop et est maintenant activement utilisé, mais il n'a pas encore été proposé comme NIP formel. Les événements Verdict (kind `7321`) et les blocs `suggestion` restent en projet et n'ont pas encore été livrés. Les retours d'implémentation de GitWorkshop et ngit détermineront si les formes deviennent un NIP autonome de revue git ou restent une convention d'application superposée sur NIP-34.

- **Noyau Nostr mail et Nostrmon** : Deux nouveaux projets de NIPs personnalisés ont circulé cette semaine. [Nostr mail core](https://njump.me/57d11cdf2f9ed73f7f39d6a7a6012ee3d642584ab11887f96a031f7d00fd9697) propose le kind `1301` pour le contenu d'e-mail RFC 2822, enveloppé avec NIP-59 pour la livraison privée et ponté vers l'e-mail legacy via des pubkeys de pont résolues par NIP-05. [Nostrmon](https://njump.me/5e9a8cee19d464f5f0322518ac9ccaf2399c69da6572346b4fb12d36acb17a27) esquisse des kinds d'événements adressables pour les régions, les cartes, les créatures, les PNJ, les sauvegardes de joueurs et les objets. Les deux restent des projets personnalisés, pas des NIPs fusionnés.

- **NIP-67 : Indice de complétude EOSE** ([PR #2317](https://github.com/nostr-protocol/nips/pull/2317)) : La proposition continue d'itérer sur l'ajout d'un marqueur de complétude positif à `EOSE`, permettant aux relais de distinguer « événements stockés entièrement livrés » des cas EOSE legacy où le relais ne fait aucune revendication de complétude.

## Six avrils Nostr

Avril offre une coupe transversale nette du chemin de développement de Nostr : le document de protocole en 2021, les premiers travaux clients en 2022, la vague d'applications post-Damus en 2023, la messagerie privée et le travail git-over-Nostr en 2024, Blossom et le nettoyage des listes de relais en 2025, et les subventions clientes axées sur l'adoption en 2026.

### Avril 2021 : le document de protocole avant le dépôt NIPs

Fiatjaf a publié l'article Nostr original, [« Notes and Other Stuff Transmitted by Relays »](https://fiatjaf.com/nostr.html), le 20 novembre 2020. Ce premier texte contenait déjà la forme centrale qui définit encore le protocole : les utilisateurs signent des événements avec des clés, les publient sur des relais, et lisent depuis les relais qu'ils choisissent. Le [journal de commits `nostr-protocol/nostr`](https://github.com/nostr-protocol/nostr/commits?since=2021-04-01&until=2021-04-30) ne montre aucun commit entre le 1er et le 30 avril. L'activité se trouve de chaque côté : les commits de mars 2021 ont ajouté les premiers liens « nostwitter » et un filtre `kind`, tandis que mai 2021 a réutilisé NIP-02 et ajouté la paternité des NIP.

En avril 2021, il n'y avait pas de marché client public, pas de réseau de relais visible, et pas de dépôt NIPs. Le protocole vivait encore comme un petit document et quelques expériences. Nostr n'était pas encore devenu un réseau social ou une plateforme de développement. C'était encore un modèle relais/clé/événement attendant sa première vague de contributeurs soutenue.

### Avril 2022 : les NIPs vivaient encore dans le dépôt principal

Avril 2022 fut le dernier mois avant que les NIPs ne sortent du dépôt principal `nostr-protocol/nostr`. Comme la scission n'avait pas encore eu lieu, le dépôt dédié [`nostr-protocol/nips`](https://github.com/nostr-protocol/nips) n'avait pas d'historique de pull requests en avril. Dans le dépôt principal, trois commits d'avril ont atterri : [« Update readme to add nip12 »](https://github.com/nostr-protocol/nostr/commit/bae286312a233b971bee5429adda7aff41747eb8) le 8 avril par goswami1999, [« add kinds list »](https://github.com/nostr-protocol/nostr/commit/4b9e9d123273ba8a5c70d77df46922070c11c11d) le 25 avril par jb55, et [« add js formatting to sample code »](https://github.com/nostr-protocol/nostr/commit/759997657f07e0344064228ffe5e93febe85d367) le 28 avril par steliosrammos.

Le travail client commençait aussi à prendre forme. Les commits Damus d'avril 2022 ont ajouté le comportement précoce des salles de chat, la gestion des profils et les icônes d'application, tandis que nostr-tools devenait le chemin de bibliothèque JavaScript pour les premiers clients et expériences. Côté protocole, les requêtes de balise génériques NIP-12 ont donné à la recherche de balise une place documentée, la liste des kinds a fait progresser Nostr vers un modèle de registre, et de meilleurs exemples JavaScript ont rendu la spécification plus facile à implémenter pour les auteurs de clients et de bibliothèques. Le 1er mai, fiatjaf a déplacé les NIPs dans le dépôt dédié. Avril 2022 fut le dernier mois de l'ère originale mono-dépôt.

### Avril 2023 : expansion des applications post-Damus

Avril 2023 est arrivé trois mois après que Damus a lancé sur l'App Store iOS le 31 janvier 2023, et après que Jack Dorsey ait publié sa clé publique Nostr. Le réseau venait d'absorber sa première grande vague de croissance publique. Les clients tels que Damus, Snort, Iris, Coracle et Amethyst étaient actifs, tandis que les opérateurs de relais apprenaient ce qu'un graphe social plus grand faisait aux hypothèses de bande passante, spam, recherche et modération.

Avril 2023 a eu une PR NIPs fusionnée : [PR #456](https://github.com/nostr-protocol/nips/pull/456), fusionnée le 17 avril, ajoutant des liens d'entités bech32 NIP-19 à la gestion d'URI NIP-21. Les commits environnants montrent la pression d'application derrière le travail de protocole. Avril 2023 a vu des travaux sur [NIP-45 COUNT](https://github.com/nostr-protocol/nips/commit/8b39976e78f90fe766ad7149e250777cddacbb5e), les marqueurs de zap spécifiques à l'événement, [NIP-15 marketplace](https://github.com/nostr-protocol/nips/commit/bf0a0da6a48b96467172414d8e41dc72b0ca379c), la sémantique de délégation de suppression NIP-26, les métadonnées de fichier NIP-94, la gestion des erreurs de wallet-connect NIP-47, et [NIP-30 emoji personnalisé](https://github.com/nostr-protocol/nips/commit/e91ce3409e1ce8267fc07a21784d2538621267c3). La liste des contributeurs s'était élargie pour inclure fiatjaf, staab, pablof7z, Semisol, CodyTseng, sethforprivacy, mikedilger, AsaiToshiya, alexgleason, martindsq, frbittencourt et arkin0x.

Damus, Snort, Iris, Coracle et Amethyst n'étaient plus des démos autour d'une spécification ; c'étaient des clients de production traitant l'onboarding, les fils, le spam, les zaps, les médias et la sélection de relais. Le travail de protocole d'avril 2023 se lit comme le backlog que ces clients ont créé : zaps, marketplaces, métadonnées de fichier, comptage, emoji et liens d'identité ont tous poussé la spécification au-delà des simples notes et suivis.

### Avril 2024 : messagerie privée, git-over-Nostr et support des mainteneurs

Avril 2024 a eu deux fusions de PR NIP. [PR #1167](https://github.com/nostr-protocol/nips/pull/1167), fusionnée le 10 avril, a corrigé une terminologie confuse dans la signature à distance [NIP-46](/fr/topics/nip-46/), où les clients et les signataires ont besoin d'un langage exact pour les actions demandées et autorisées. [PR #1108](https://github.com/nostr-protocol/nips/pull/1108), fusionnée le 17 avril, a étendu les dépôts git [NIP-34](/fr/topics/nip-34/) avec des événements de statut, des clarifications, des mainteneurs optionnels, des identifiants de dépôt et des balises de découvrabilité. Cette étape a rendu git-over-Nostr plus pratique pour ngit et plus tard GitWorkshop.

[NIP-17](https://github.com/nostr-protocol/nips/commit/df30012430c88d49fb5b124992b04d5c61b6338b), anciennement NIP-24, a atterri le 24 avril en tant que messages gift-wrapped scellés pour les DMs privés et les petits chats de groupe. Le travail sur les clients et bibliothèques a suivi : Amethyst, Primal, Gossip, nostr-tools, NDK et rust-nostr étaient tous actifs dans la même période.

OpenSats a également annoncé un support à long terme pour les développeurs Nostr en avril 2024 : [PabloF7z](https://opensats.org/blog/pablofz7-receives-lts-grant) le 9 avril, [Stuart Bowman](https://opensats.org/blog/stuart-bowman-receives-lts-grant) le 12 avril, et [hzrd149](https://opensats.org/blog/hzrd149-receives-lts-grant) le 15 avril. Ces subventions ont déplacé le financement des subventions de projets isolés vers la maintenance soutenue des relais, bibliothèques et infrastructure client.

### Avril 2025 : nettoyage dense des NIPs et formalisation de Blossom

Avril 2025 fut le mois de protocole le plus dense de cette rétrospective, avec seize PRs NIPs fusionnées. Le mois a commencé avec [PR #1846](https://github.com/nostr-protocol/nips/pull/1846), ajoutant les transactions et adresses blockchain à NIP-73, et [PR #1865](https://github.com/nostr-protocol/nips/pull/1865), ajoutant les balises NIP-C0 à la table des balises standardisées. Il a continué avec [PR #1801](https://github.com/nostr-protocol/nips/pull/1801) et [PR #1889](https://github.com/nostr-protocol/nips/pull/1889), améliorant tous deux les directives de republication de liste de relais de kind `10002`, et [PR #1879](https://github.com/nostr-protocol/nips/pull/1879), qui a rétréci et clarifié [NIP-65](/fr/topics/nip-65/).

[PR #1822](https://github.com/nostr-protocol/nips/pull/1822) a ajouté NIP-B7 pour l'interaction Blossom, donnant aux clients Nostr et aux serveurs Blossom une couche de coordination canonique après plus d'un an de pratique informelle. [PR #1051](https://github.com/nostr-protocol/nips/pull/1051) a déprécié [NIP-26](https://github.com/nostr-protocol/nips/blob/master/26.md), la spécification de signature d'événement déléguée. NIP-26 avait été difficile à implémenter en toute sécurité et était devenu moins attractif à mesure que NIP-46 et d'autres modèles de signataire mûrissaient.

Le reste du mois a combiné nettoyage et expansion d'application : [PR #1882](https://github.com/nostr-protocol/nips/pull/1882) a ajouté des champs politique de confidentialité et conditions d'utilisation à [NIP-11](/fr/topics/nip-11/), [PR #1849](https://github.com/nostr-protocol/nips/pull/1849) a étendu les signets web de kind `39701` sous NIP-B0, [PR #1891](https://github.com/nostr-protocol/nips/pull/1891) a ajouté ce kind de signet au README, et [PR #1895](https://github.com/nostr-protocol/nips/pull/1895) a ajouté les balises standardisées NIP-B0. OpenSats a annoncé sa [Onzième Vague de Subventions Nostr](https://opensats.org/blog/eleventh-wave-of-nostr-grants) le 16 avril, finançant Swae, HAMSTR, Vertex, Nostr Double Ratchet et Nostr Game Engine. Primal, Coracle, noStrudel, nostr-tools, NDK et rust-nostr livraient aussi pendant cette période, donc le nettoyage du protocole se situait à côté du travail actif sur les clients et bibliothèques.

### Avril 2026 : durcissement de NIP-34, badges et subventions axées sur l'adoption

Avril 2026, le mois où ce numéro se termine, a eu quatre PRs NIPs fusionnées. La première fut [PR #2276](https://github.com/nostr-protocol/nips/pull/2276), fusionnée le 1er avril, qui a changé les badges de profil [NIP-58](https://github.com/nostr-protocol/nips/blob/master/58.md) au kind `10008` et ajouté les ensembles de badges de kind `30008`, rendant l'attribution de badges et les collections de badges plus composables. Un deuxième changement de convivialité git-over-Nostr est arrivé dans [PR #2312](https://github.com/nostr-protocol/nips/pull/2312), fusionnée le 10 avril, ajoutant la sémantique d'URL de clone `nostr://` à [NIP-34](/fr/topics/nip-34/). Les nettoyages du 25 avril, [PR #2325](https://github.com/nostr-protocol/nips/pull/2325) et [PR #2326](https://github.com/nostr-protocol/nips/pull/2326), ont supprimé un langage inutilisé et incorrect de NIP-34.

Les commits connexes affinent les mêmes surfaces. Le 22 avril, fiatjaf a ajouté une liste de serveurs Blossom à NIP-51 et ajusté l'édition des métadonnées NIP-29 pour correspondre au comportement de style PUT de Flotilla. Le 26 avril, il a renommé NIP-5A pour plus de clarté. Avril 2026 s'est concentré sur rendre les surfaces de protocole déjà utilisées plus faciles à implémenter et plus difficiles à mal interpréter.

OpenSats a annoncé sa [Seizième Vague de Subventions Nostr](https://opensats.org/blog/sixteenth-wave-of-nostr-grants) le 8 avril, soutenant Amethyst Desktop, Nostr Mail, Nostrord, Nurunuru (null--nostr) et un renouvellement HAMSTR : clients de bureau, messagerie de type e-mail, UX de groupe, onboarding japonais et connectivité hors réseau.
